# 完整 App 故事驾驶舱 UX 设计说明

## 源文件与生成物边界

- 设计规格源文件：`docs/superpowers/specs/2026-05-12-complete-app-story-cockpit-ux-design.md`。
- OpenSpec 主记录：`openspec/changes/design-complete-app-story-cockpit-ux/`。
- 本 change 不修改 `dist/`，不修改运行时代码，不生成前端产物。

## 核心取舍

首版完整 App 不应从“编辑器优先”或“审阅队列优先”开始，而应采用“故事驾驶舱居中”。原因是 StorySpec 的核心问题不是单点编辑，而是作者、团队、Agent 和正典门禁之间的状态可理解性。

故事驾驶舱负责回答：

- 当前故事是什么。
- 当前卡点是什么。
- 下一步最该做什么。
- 哪些候选、冲突、任务或审阅需要处理。
- 最近团队和 Agent 做了什么。

## 页面边界

- 故事驾驶舱：导向和状态聚合，不承载重编辑。
- 章节与写作：推进 outline → scene → sample → draft → review，不直接吞并正典审阅。
- 候选与正典审阅：处理高影响内容的预览、确认、应用和来源追踪。
- 任务中心：解释 Agent run、人工任务、失败、阻塞和跳转。

## Preview / Confirm / Apply 边界

涉及正典事实、世界观、角色关系、主题、感情推进或章节关键事件时，界面必须默认进入 Preview / Confirm / Apply。Agent run 和团队建议只能提交候选、预览或评论，不能无声写入正式故事。

后续实现可以为低风险个人偏好提供快速应用，但不得扩大到正典和高影响内容。

## 视觉边界

视觉采用“工作室控制台”：浅色、清晰、可扫描，保留创作产品的温度。避免营销页 Hero、装饰性渐变、层层套卡片、过重暗色后台和只靠技术词解释流程。

## 后续拆分建议

后续应先做 API contract 与前端状态模型，再进入实现：

1. App shell 与路由骨架。
2. 故事驾驶舱数据模型和 UI。
3. 章节与写作通道。
4. 候选与正典审阅。
5. 任务中心。
6. 权限、活动流、资料库和设置的第二批设计/实现。
