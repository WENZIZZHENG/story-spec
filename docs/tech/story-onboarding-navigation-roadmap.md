# 新故事引导与创作导航路线图

## 状态

Planned。本文记录下一阶段“新故事引导与创作导航”的开发待办；尚未开始实现。当前执行入口见 [todo-index.md](todo-index.md)。

## 背景

上一轮创作控制权优化已经让命令更倾向于先澄清、给示例、写入前预览。但用户从一句话创意进入项目时，仍容易遇到两个体验断点：

- 用户只给出类型、基调和少量关键词时，系统可能过早扩写大量设定，导致用户创造空间被压缩。
- 新用户需要理解较多命令和目录概念，才知道下一步应该继续澄清、生成规格预览、检查项目，还是开始拆任务。

本路线把重点放在“新故事从哪里开始”和“下一步怎么走”，让系统成为创作导航，而不是替用户一次性定稿。

## 目标

- 为新小说提供清晰入口：一句命令创建故事创意草稿，并引导进入澄清访谈。
- 在任意阶段提供下一步导航：告诉用户当前缺什么、可以做什么、哪些内容还不适合写入正稿。
- 强化澄清校验：阻止未确认的 AI 建议被直接提升为规格、任务或正文依据。
- 让用户能看见创作控制权状态：哪些是用户已确认，哪些只是 AI 候选，哪些可能发生偏离。
- 改善访谈体验：更多“需要澄清”、更多 2-3 个可复制示例，同时保留“不知道 / 稍后决定 / 给我示例”等低压力选择。

## 非目标

- 不开发旧项目迁移助手。
- 不兼容已废弃的旧项目结构；后续只服务当前 `stories/<story>/...` 结构。
- 不把一句话创意自动扩写成完整世界观、人物谱系或章节大纲。
- 不新增 GUI；本路线先覆盖 CLI、命令提示和文档路径。
- 不把项目改成纯种田、纯模板生成或自动代写流程；建设流、思想改造等只能作为具体作品的支撑工具，而不是通用产品默认倾向。

## 设计原则

- **先问再写**：创意信息不足时，优先提出澄清问题和示例分支。
- **示例是脚手架，不是答案**：示例必须允许用户复制、改写或拒绝，不能被默认当成 canon。
- **下一步可见**：每个阶段都能输出清楚的下一步建议。
- **确认才晋级**：AI 补全、候选分支、风格建议，只有被用户确认后才能进入规格、任务或正文。
- **降低术语门槛**：用户不需要先理解内部目录和 SDD 细节，也能知道该怎么继续。

## 优先级视图

| 优先级 | 批次 | 主题 | 说明 |
| --- | --- | --- | --- |
| P0 | E0 | `novel story:new` | 解决新故事从哪里开始 |
| P0 | E1 | `novel next` | 解决当前下一步是什么 |
| P0 | E2 | 澄清校验门禁 | 阻止未确认建议进入下游 |
| P1 | E3 | 创作控制权报告 | 让确认状态、待澄清项和漂移风险可见 |
| P1 | E4 | 访谈体验增强 | 让问题更少重复、更有分叉、更容易回答 |
| P2 | E5 | CLI preview/apply | 把写入前预览流程沉淀为稳定 CLI 能力 |

## Batch E0：新故事入口

新增 `novel story:new <name>`。

范围：

- 创建 `stories/<story>/idea.md`。
- 支持 `--idea "一句话创意"`，把用户原始输入记录为“用户原文”，不做未经确认的扩写。
- 创建后提示运行 `novel interview <story>`，并说明这一步会先澄清再生成规格预览。
- 若故事名已存在，明确提示选择新名称或进入现有故事。

验收口径：

- 输入 `novel story:new 法术编译纪元 --idea "异界穿越、轻松冒险、编程施法、慢热感情、文明级威胁"` 后，只生成创意草稿和下一步提示，不生成完整设定。
- `idea.md` 能区分用户原文、待澄清问题、AI 候选建议。
- 单元测试覆盖新建成功、重复名称、空 idea、带 `--idea` 四类路径。

## Batch E1：下一步导航

新增 `novel next [story]`。

范围：

- 根据故事状态输出当前最合适的 2-4 个下一步动作。
- 能识别以下状态：只有 idea、澄清未完成、已有规格预览未确认、规格已确认但任务未拆、任务已拆但未写作、已有正文但需要 review。
- 输出建议时附带可复制命令，例如 `novel interview <story>`、`novel preview specify <story>`、`novel review <story>`。

验收口径：

- 新故事创建后，`novel next <story>` 首选继续澄清，而不是立即写正文。
- 存在未确认 AI 候选时，`novel next` 明确提示先确认或拒绝。
- 状态缺失或结构异常时，给出修复建议，不直接吞掉错误。

## Batch E2：澄清校验门禁

增强 `novel validate` 的澄清相关检查。

范围：

- 校验 `clarifications.json` 是否存在、可解析、字段完整。
- 检查必答项是否仍未回答。
- 检查未确认的 AI 建议是否被规格、任务、章节正文引用。
- 把违规项归类为 blocking / warning，方便后续命令决定是否继续。

验收口径：

- 未确认候选被写入 `specification.md` 时，`novel validate` 能报出来源和目标文件。
- 用户明确标记“稍后决定”的问题不会被误判为已确认设定。
- 测试覆盖有效澄清、缺失澄清、JSON 损坏、未确认建议被引用四类路径。

## Batch E3：创作控制权报告

新增 `novel creative:report [story]`。

范围：

- 输出用户已确认内容、仍需澄清内容、AI 候选建议、可能偏离用户原意的内容。
- 报告中给出下一步建议，但不直接写入故事文件。
- 后续可考虑让 `novel status --creative` 作为别名或摘要入口；本批次先实现独立命令。

验收口径：

- 报告能清楚区分“用户确认”和“AI 候选”。
- 对缺少澄清文件的项目，报告能提示先运行访谈或创建澄清记录。
- 输出格式适合终端阅读，并提供可复制的后续命令。

## Batch E4：访谈体验增强

继续优化 `novel interview`。

范围：

- 根据已回答内容减少重复问题。
- 每个关键问题提供 2-3 个可复制示例，且示例之间必须代表不同创作方向。
- 对困难问题提供固定选择：`不知道`、`稍后决定`、`给我示例`。
- 支持基础依赖问题，例如用户选择“编程施法”后，再追问运行时、代价、错误类型、调试方式等，而不是泛泛追问魔法体系。

验收口径：

- 对“异界穿越、轻松冒险、编程施法、慢热感情、文明级威胁”这类输入，访谈优先澄清冲突和边界，不直接产出大段世界观。
- 示例能覆盖至少 3 种明显不同方向，例如轻喜剧冒险、学院调查、边境建设冒险。
- 已回答字段不会在同一轮被原样重复询问。

## Batch E5：CLI preview/apply

把写入前预览能力沉淀为稳定 CLI 路径。

范围：

- 新增 `novel preview specify <story>`，生成规格预览，不直接覆盖正式规格。
- 新增 `novel apply <preview-id>`，只在用户确认后写入。
- 预览记录需要包含来源摘要、待确认项和写入目标。

验收口径：

- 预览失败不会污染正式故事文件。
- `apply` 只能应用存在且未过期的预览。
- 未确认澄清项存在时，预览必须标记风险，必要时阻止应用。

## 风险与缓解

| 风险 | 缓解 |
| --- | --- |
| 新命令过多，用户仍然不知道用哪个 | 用 `novel next` 做导航入口，所有新命令都能从它被发现 |
| 示例变成隐性默认答案 | 在数据结构中区分用户确认、AI 候选、待确认，不把示例直接写入 canon |
| 校验过严导致用户无法试写 | 区分 blocking / warning；只阻止进入正式规格或任务，不阻止临时草稿 |
| CLI preview/apply 牵涉写入机制较深 | 放到 P2，先完成 story:new、next 和 validate 的核心链路 |

## 完成后需要同步

- 更新 [todo-index.md](todo-index.md) 的状态或移除已完成路线。
- 更新 [todo-archive.md](todo-archive.md)，记录完成范围和验证证据。
- 若涉及 CLI 行为、模板契约、生成产物或项目结构变化，新增 `changes/YYYY-MM-DD-story-onboarding-navigation.md`。
- 更新用户文档中的快速开始、命令说明或工作流文档。
- 至少运行 `git diff --check`；涉及代码时按批次补充单元测试和 `npm test` / `npm run build` 等验证。
