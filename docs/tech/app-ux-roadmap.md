# 完整 App 产品体验路线图

## 状态

Active。本文承接完整 App 的产品体验、信息架构和界面设计任务。P1-0 的核心设计决策已确认并记录到 [完整 App 故事驾驶舱体验设计](../superpowers/specs/2026-05-12-complete-app-story-cockpit-ux-design.md)。它是后续前端架构、API 契约和多人平台实现的产品输入，不表示完整 Web App 已经实现。

## 当前主线

先让用户知道“从哪里开始、当前故事卡在哪里、哪些内容只是候选、什么动作会写入正典”，再讨论前端框架和视觉实现。当前 `storyspec app` 是实验性本机工作台，不能简单美化后当作最终多人在线 App。

## 覆盖功能缺口

- 完整前端 App 架构：路由、App shell、登录态、权限态、空状态、错误状态、全局导航、API client 和 E2E。
- 写作编辑器：章节草稿编辑、自动保存、本地缓存、上下文面板、审稿反馈、版本差异和移动端边界。
- 项目与故事工作台：项目首页、故事档案、章节总览、大纲视图、任务板、伏笔看板、关系图、世界观资料库、搜索和命令面板。
- 离线优先与本地缓存：断网写作、本地草稿、冲突恢复和“离线只能写候选，不自动 apply”的边界。

## P1-0 完整 App 产品体验与界面重设计

- 类型：产品体验、信息架构、UI/UX 设计
- 当前设计规格：[完整 App 故事驾驶舱体验设计](../superpowers/specs/2026-05-12-complete-app-story-cockpit-ux-design.md)
- 已确认设计决策：首版采用“作者 + 团队平衡”，信息架构为“故事驾驶舱居中”，项目/工作区入口作为 App shell 必备入口，故事内主页面为故事驾驶舱、章节与写作、候选与正典审阅、任务中心；高影响内容默认走 Preview / Confirm / Apply；视觉方向为“工作室控制台”。
- 首批实现切片：已通过 `add-complete-app-story-cockpit-first-slice` 建立 complete App state contract、本机 App 状态 endpoint 和工作室控制台 shell；`add-complete-app-frontend-architecture-slice` 进一步抽出首批 route/API/status contract 和本机 shell API 地图；`add-local-app-guided-first-run` 已在本机 Web 首屏补三步开始路径，降低“打开后不知道怎么用”的摩擦。后续仍需真实多人产品流、独立前端项目、富文本编辑器和实时协作。
- 背景/问题：当前 `storyspec app` 仍是实验性本机工作台，页面信息层级、入口说明和操作反馈不足；用户打开 Web 后不容易判断“下一步该做什么”“哪些内容只是候选”“哪些动作会写入正式故事”。如果直接进入前端架构重写，容易把不清晰的产品模型固化到路由、API 和数据模型里。
- 已有基础：`src/app-server/local-app-html.ts`、本机 App API、server 控制面 API、preview / confirm / apply 流程、已归档的新用户引导和创作导航路线、归档设计能力 `ui-ux-designer` / `frontend-design` / `ui-ux-pro-max`。
- 缺口：产品定位、用户旅程、信息架构、主导航、页面地图、首批状态语言和本机 Web 首屏开始路径已完成首批落地；仍缺完整独立前端、真实登录/权限 UI、协作评论审批 UI、富文本编辑器和实时协作体验。
- 建议方案：
  1. 先产出产品体验设计规格，再进入前端架构和 API 契约实现。
  2. 明确核心角色和使用路径：作者、编辑、审稿者、只读成员、agent-runner 分别如何创建项目、进入故事、处理候选、评论、审批和应用正典。
  3. 设计完整 App 信息架构：工作区、项目、故事、章节、设定库、任务、审稿、成员、设置。
  4. 先画低保真页面结构和交互流：Dashboard、Project Workspace、Writing Editor、Canon Review、Agent Task Center、Team / Permissions。
  5. 定义视觉方向：专业创作工作台、编辑部协作、清晰状态面板；避免把 App 做成营销首页、装饰性卡片堆叠或只靠技术词解释流程。
  6. 制定状态和空状态文案，把候选、预览、试运行、应用、阻塞、稍后决定等状态统一成可读、可审计、可回退的用户语言。
- 交付物：
  1. `完整 App 产品体验设计规格`：说明产品定位、目标用户、核心路径、主导航、页面地图和非目标。
  2. `用户旅程图`：覆盖首次进入、继续创作、多人审阅、agent 任务、正典应用和冲突处理。
  3. `App 信息架构 / 页面地图`：标出工作区、项目、故事、章节、设定库、任务、审稿、成员和设置之间的进入关系。
  4. `核心页面低保真结构`：至少覆盖项目列表、故事工作台、章节写作入口、候选审阅、任务中心和成员权限。
  5. `状态语言词典`：把技术状态映射成界面文案、按钮文案、风险提示和下一步动作。
  6. `P1-4 前端架构输入清单`：列出必须由 API 契约支持的页面数据、权限状态、错误状态和加载状态。
- 首批页面范围：
  1. 第一批必须设计：项目列表 / 工作区首页、故事工作台、章节写作入口、候选审阅、任务中心。
  2. 第二批跟进设计：成员权限、设定库、活动流、项目设置。
  3. 暂缓设计：计费、公开社区、插件市场、完整移动端、完整实时协同编辑器。
- 用户走查场景：
  1. 新用户第一次打开 App，能知道从哪里创建或继续一个故事。
  2. 作者看到 AI 生成内容时，能分清“候选方案”和“已写入正典”。
  3. 编辑或审稿者能知道自己当前能评论、能审批，还是只能查看。
  4. 任务失败、权限不足、内容冲突或流程阻塞时，界面能说明原因和下一步，而不是只显示失败。
  5. 用户能从工作台判断当前故事卡在哪里、最近谁做了什么、哪些内容等待确认。
- 状态语言词典范围：
  1. `candidate`：候选方案，必须显示来源、生成任务、适用范围和是否可应用。
  2. `preview`：预览变更，必须说明不会自动写入正式故事。
  3. `dry-run`：试运行，必须说明只检查结果和影响，不执行写入。
  4. `apply`：应用到正式故事，必须带确认、diff、风险提示和回滚入口。
  5. `blocked`：暂时无法继续，必须说明阻塞原因和推荐处理动作。
  6. `deferred`：稍后决定，必须允许用户保留未确认状态，不被系统当作完成。
  7. `canon`：正典 / 已确认事实，必须和候选、草稿、评论视觉区分。
- 涉及文件/模块：`src/app-server/local-app-html.ts`、`src/app-server/local-app-server.ts`、`src/server/http/*`、未来 `app/` 或 `web/`、后续设计规格文档。
- 验收标准：设计规格覆盖目标用户、核心路径、主导航、页面地图、关键状态、空状态、权限反馈和响应式边界；交付物清单全部可定位；首批页面都有低保真结构、主要状态和入口说明；用户走查场景都能用文字或线框完整走通；一个没读过文档的新用户能在 3 分钟内完成创建故事、找到章节入口、理解候选不会自动写入、知道在哪里审阅并应用结果；完整前端架构任务必须引用本任务产物。
- 参考项目/资料：当前本机 App 流程和已归档 StorySpec 首程引导路线；归档 `ui-ux-designer`、`frontend-design`、`ui-ux-pro-max` 只作为设计方法和可访问性检查参考，不直接决定前端技术栈。
- 不做/边界：本任务不写业务代码、不选择最终前端框架、不实现实时协同或富文本编辑器；不把当前本机 Web 简单美化成最终产品；不承诺实时协同、富文本、云端 SaaS 已完成；不把未确认候选内容展示成正式成果；不为了视觉效果牺牲 preview / confirm / apply 的作者控制权。

## P2-1 写作编辑器体验评估

- 类型：编辑器、研究、用户体验
- 背景/问题：完整 App 需要浏览器端编辑体验，但 StorySpec 当前产物以 Markdown、YAML、JSON 和任务文档为主；直接上富文本可能破坏可审计产物。
- 已有基础：章节草稿、draft promote dry-run、compile manuscript、style lint、review loop。
- 缺口：缺少浏览器端章节草稿编辑器、自动保存、本地草稿缓存、恢复未保存内容、编辑锁/冲突提示、Markdown AST/ProseMirror schema 映射和结构化片段同步策略。
- 建议方案：先做研究 OpenSpec，对比纯 Markdown editor、ProseMirror/Tiptap、Monaco/CodeMirror、结构化表单四种方案；第一阶段只允许编辑草稿和评论，不直接编辑正式正典。
- 涉及文件/模块：未来 web editor、`src/application/manage-drafts.ts`、`src/application/compile-manuscript.ts`、style/review modules。
- 验收标准：选型报告明确 schema、存储、协作、导出和测试策略；至少一个章节草稿编辑 PoC 不污染正式 content。
- 参考项目/资料：Tiptap、ProseMirror、Yjs 官方文档。
- 不做/边界：不在研究阶段承诺最终编辑器依赖；实时编辑文档只是草稿态，不把 CRDT 文档直接当正典。

## P3-1 离线优先、PWA 与本地缓存

- 类型：研究、前端体验
- 背景/问题：小说写作存在断网、本地保存和长时间编辑需求；完整 App 若只依赖实时连接，体验会脆弱。
- 建议方案：研究 IndexedDB、本地草稿缓存、冲突恢复、导出备份和“离线只能写候选，不自动 apply”的边界。
- 验收标准：输出研究记录和 PoC 计划；明确离线写入不会绕过正典门禁。
- 不做/边界：不在多人基础未稳前实现完整离线同步。
