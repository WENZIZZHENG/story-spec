# 完整 App 故事驾驶舱体验设计

## 状态

已确认设计，等待用户 review 后进入实施计划。本文是“多人在线写作平台 / 完整 App”的产品体验输入，不表示完整 Web App、账号系统、云端数据库、实时协作或富文本编辑器已经实现。

## 设计背景

当前 `storyspec app` 是实验性本机工作台。它已经能承载一部分本地流程，但页面信息层级、入口说明和操作反馈不足。用户打开 Web 后容易不知道从哪里开始、当前故事卡在哪里、哪些内容只是候选、哪些动作会写入正式故事。

完整 App 的第一版应先解决“可理解、可推进、可审计”的体验问题，再进入前端架构、API 契约、数据库和实时协同实现。

## 已确认决策

- 首版产品优先级：作者 + 团队平衡。
- 信息架构方向：故事驾驶舱居中。
- 首批页面范围：项目/工作区入口作为 App shell 必备入口；故事内主页面为故事驾驶舱、章节与写作、候选与正典审阅、任务中心。
- 核心交互门槛：默认使用 Preview / Confirm / Apply。
- 页面职责边界：驾驶舱负责导向，写作页负责推进文本，正典页负责确认事实，任务中心负责解释 Agent 和团队任务状态。
- 视觉方向：工作室控制台。

## 产品定位

StorySpec 完整 App 是一个面向长篇故事创作的协作工作台。它不是单纯的富文本编辑器，也不是只展示任务日志的工程后台。

首版 App 要帮助用户完成五件事：

1. 知道当前在哪个工作区、项目和故事里。
2. 知道当前故事卡在哪里，下一步最该做什么。
3. 知道哪些内容是候选、草稿、评论、预览或已确认正典。
4. 知道团队成员和 Agent 最近做了什么，以及是否需要自己处理。
5. 知道高影响变更何时会写入正式故事，并能在写入前预览和确认。

## 目标用户

- 作者：拥有故事方向、正典事实和最终确认权，关注连续写作、下一步动作、候选质量和冲突风险。
- 编辑：帮助整理设定、章节节奏和审阅意见，关注候选审阅、评论、影响范围和任务状态。
- 审稿者：提出批注和修改建议，关注章节草稿、评论上下文和审批边界。
- 只读成员：查看项目状态、章节进展和已确认事实，不能直接修改高影响内容。
- Agent runner：提交结构化候选、检查结果、任务状态和失败原因，不能绕过预览确认直接写入正典。

## 信息架构

首版采用稳定 App shell：

- 左侧全局导航：项目与故事、故事驾驶舱、章节与写作、候选与正典、任务中心、资料库、团队与权限、设置。
- 中间主工作区：根据当前页面呈现故事状态、章节通道、正典审阅或任务详情。
- 右侧协作侧栏：根据上下文显示审阅队列、成员动态、Agent 任务、阻塞原因和相关评论，可在窄屏收起。

### 首批页面

0. 项目/工作区入口
   - App shell 必备入口，不作为故事内主工作面。
   - 支持创建项目、打开已有项目、查看最近项目和选择当前故事。
   - 没有项目或没有当前故事时，必须把用户导向可执行的创建/打开动作。

1. 故事驾驶舱
   - 默认首页。
   - 展示当前故事、当前卡点、下一步动作、章节进度、故事健康度、打开的决策和协作侧栏摘要。
   - 主动作包括继续写作、处理候选、查看 Agent 结果、进入任务详情。

2. 章节与写作
   - 承载 outline → scene → sample → draft → review 的章节通道。
   - 首版不追求复杂富文本，而是先让章节流程、样章、草稿、审稿反馈和候选入口串起来。
   - 高影响内容不直接写入正典，必须转入候选或预览。

3. 候选与正典审阅
   - 集中处理设定、角色关系、世界观事实、章节关键事件和冲突。
   - 每个候选必须展示来源、影响范围、相关章节、风险提示和可执行动作。
   - 主动作包括接受、驳回、请求修改、查看影响范围和追踪来源。

4. 任务中心
   - 统一展示 Agent run、人工任务、失败原因、阻塞状态和排队状态。
   - 主动作包括查看结果、重试、取消、转为候选、跳到相关章节。
   - 不让任务状态散落在 CLI 输出、日志或不可见后台里。

### 第二批页面

- 团队与权限：成员、角色、邀请、只读/审阅/审批能力和权限不足反馈。
- 资料库：世界观、人物、地点、术语、参考资料和搜索。
- 活动流：团队与 Agent 的可审计时间线。
- 项目设置：项目元数据、方法预设、导入导出和安全边界。

### 暂缓范围

- 计费。
- 公开社区。
- 插件市场。
- 完整移动端体验。
- 完整实时协同编辑器。
- 云端 SaaS 部署能力。

## 核心流程

### 作者继续写作

1. 作者打开故事驾驶舱。
2. 驾驶舱显示当前故事、当前卡点和下一步动作。
3. 作者进入章节与写作。
4. 章节通道显示当前阶段：outline、scene、sample、draft 或 review。
5. 如果生成内容涉及设定、角色关系、世界观事实或章节关键事件，系统先创建候选或预览。
6. 作者确认后，内容才能进入草稿或正典。

### 团队审阅候选

1. 编辑或审稿者从协作侧栏或候选与正典审阅进入待处理项。
2. 页面展示候选来源、建议内容、影响范围、冲突和相关章节。
3. 成员可以评论、请求修改、驳回或提交给作者确认。
4. 有权限的成员确认后，系统应用变更并记录来源。

### Agent 任务处理

1. Agent run 出现在任务中心。
2. 成功任务提供结果预览和“转为候选”动作。
3. 失败任务展示失败原因、可重试条件和阻塞项。
4. 高影响 Agent 结果必须进入 Preview / Confirm / Apply，不允许静默写入正典。

### 冲突处理

1. 驾驶舱和正典审阅页突出显示冲突数量。
2. 冲突详情展示相关事实、章节、来源和推荐处理动作。
3. 用户可以接受某个版本、请求修改、暂缓决定或创建后续任务。
4. `deferred` 不被系统当作完成状态，必须保留在可见队列中。

## Preview / Confirm / Apply

首版默认门槛是 Preview / Confirm / Apply。

- Preview：展示将要改变什么，不执行写入。
- Confirm：由作者或授权成员确认。
- Apply：确认后写入草稿、正典或任务状态，并记录来源和时间。

必须走该门槛的内容：

- 正典事实。
- 世界观规则。
- 角色关系。
- 主题和感情推进。
- 章节关键事件。
- Agent 生成的高影响建议。

允许后续做快速应用的低风险内容：

- 过滤条件。
- 页面偏好。
- 草稿内临时标记。
- 不影响正典的个人视图状态。

## 状态语言词典

| 技术状态 | 界面名称 | 用户含义 | 主要动作 |
| --- | --- | --- | --- |
| `candidate` | 候选方案 | 可参考但尚未确认，不是正式故事事实 | 预览、接受、驳回、请求修改 |
| `preview` | 预览变更 | 只展示影响，不会自动写入正式故事 | 查看差异、确认、返回修改 |
| `dry-run` | 试运行 | 检查结果和影响，不执行写入 | 查看报告、修复问题、重新运行 |
| `apply` | 应用到正式故事 | 经确认后写入草稿、正典或任务状态 | 确认应用、查看来源、回滚入口 |
| `blocked` | 暂时无法继续 | 有缺失、冲突、权限或任务失败阻塞流程 | 查看原因、处理阻塞、创建任务 |
| `deferred` | 稍后决定 | 用户明确保留未确认状态 | 稍后提醒、保留在队列、重新打开 |
| `canon` | 正典 / 已确认事实 | 已被作者或授权成员确认的故事事实 | 查看来源、引用、申请修改 |
| `draft` | 草稿 | 可编辑文本，不等同于正典事实 | 保存、提交审阅、创建候选 |
| `comment` | 评论 | 审阅意见或讨论，不直接改变故事 | 回复、解决、转任务 |

## 权限与协作反馈

首版界面必须清楚告诉用户当前能做什么：

- 作者：可以确认高影响变更、应用正典、处理冲突。
- 编辑：可以提交候选、评论、请求修改；能否确认由项目权限决定。
- 审稿者：可以评论和提交修改建议，不能默认 apply 正典。
- 只读成员：只能查看，不显示会误导为可执行的主按钮。
- Agent：只提交结果、候选、检查和失败原因，不拥有最终确认权。

权限不足时，界面应说明原因和可请求的下一步，而不是只显示禁用按钮。

## 视觉方向

视觉采用“工作室控制台”：

- 浅色工作台背景，内容区域清晰，不做营销页式大 Hero。
- 蓝色用于导航、选中态和信息状态。
- 橙色用于下一步、待确认和高注意力动作。
- 绿色用于已确认、已完成和正向状态。
- 左导航稳定，中间主工作区信息密度适中，右侧协作栏可收起。
- 卡片只用于具体信息块、任务项、候选项或弹层，不把页面章节层层套成卡片。
- 避免装饰渐变、漂浮元素、过重暗色后台风和只有技术词的日志界面。

## 响应式边界

- 桌面宽屏：左导航、中间主区、右侧协作栏三栏并列。
- 中等屏幕：右侧协作栏可折叠，主区保持完整。
- 手机窄屏：左导航变为抽屉或底部主导航，协作侧栏进入独立面板；首版只保证核心查看和轻量处理，不承诺完整移动端编辑。
- 所有主要按钮需要可触达、可聚焦、可键盘操作。
- 状态文案不能依赖颜色单独表达，必须有文字标签。

## 空状态和错误状态

- 没有项目：显示创建项目和打开已有项目两个主动作。
- 没有当前故事：引导创建或选择故事。
- 没有候选：说明当前没有待确认内容，并提供继续写作或运行检查入口。
- 没有任务：说明没有正在运行或阻塞的任务，不展示空白列表。
- 任务失败：展示失败原因、输入上下文、可重试动作和相关页面入口。
- 权限不足：说明当前角色、缺少的权限和可请求的处理方式。
- 冲突存在：说明冲突影响范围和推荐处理动作。

## 后续 API / 前端契约输入

后续 P1-2 API contract 至少需要支持：

- 当前工作区、项目、故事、章节的选择态。
- 当前故事卡点、下一步动作、章节通道阶段和故事健康度。
- 候选列表、候选来源、影响范围、diff、冲突和确认状态。
- 任务列表、Agent run、失败原因、阻塞原因、重试条件和跳转目标。
- 成员角色、权限能力、只读/审阅/审批边界。
- 活动流、来源追踪、应用记录和回滚入口。
- 加载、空状态、错误状态和权限不足状态。

## 验收标准

- 一个未读过文档的新用户能在 3 分钟内找到创建或继续故事的入口。
- 用户能从故事驾驶舱判断当前故事卡在哪里、下一步做什么、哪些内容等待确认。
- 用户能分清候选、预览、草稿、评论和正典。
- 高影响内容不会绕过 Preview / Confirm / Apply。
- 团队成员能知道自己能评论、能审批，还是只能查看。
- Agent 任务成功、失败、排队、阻塞都有可理解说明和下一步动作。
- 首批四页职责清晰，不互相吞并。
- 视觉像专业创作工作台，不像营销页、日志后台或纯命令面板。

## 非目标

- 不实现完整 Web App。
- 不选择最终前端框架。
- 不实现账号、云端、计费、公开社区或 SaaS 部署。
- 不实现完整富文本编辑器。
- 不实现完整实时协同编辑器。
- 不把当前本机 Web 简单美化后宣称为最终多人在线 App。
- 不把未确认候选展示成正式成果。
