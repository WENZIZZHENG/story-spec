# Novel Writer 项目 SDD

## 状态

Active。本文档是 Novel Writer 仓库开发与重构的项目级 SDD，优先级高于全局 SDD 中与本项目冲突的规则。

## 1. 适用范围

- 适用于本仓库的代码、CLI、模板、生成产物、测试、技术文档和重构路线。
- 当前阶段目标是全量重构 Novel Writer，而不是在旧架构上做保守增量。
- `novel-sdd.md` 是产品方法论与创作理念文档，不是开发规范，不参与本仓库开发流程约束。
- `docs/tech/todo-index.md` 是当前活跃待办入口；`docs/tech/todo-governance.md` 定义待办、路线图和归档规则。
- `docs/tech/archive/full-refactor/full-refactor-*.md` 是已完成重构路线图与历史任务索引；新开发应先从统一待办入口找到当前路线，再按对应专题文件推进。

## 2. 优先级

遇到冲突时按以下顺序执行：

1. 用户最新明确指令。
2. 本文件 `SDD.md`。
3. `docs/tech/todo-index.md` 与当前专题路线文件中的任务目标。
4. 仓库 `AGENTS.md`。
5. 全局 SDD。

全局 SDD 只作为兜底流程参考；凡是本文件已经规定的事项，以本文件为准。

## 3. 重构原则

- 以目标架构为中心，不以旧入口、旧 CLI 参数、旧模板格式或旧测试快照为中心。
- 旧功能不默认保留；只有路线图、当前任务或用户明确要求兼容时，才维护旧行为。
- 允许破坏性变更，但必须在技术文档、测试或迁移说明中清楚记录影响。
- 优先完成清晰的纵向切片，而不是为了最小 diff 保留过时抽象。
- 可以删除、合并或重命名旧模块；删除前应确认它不再属于目标架构，或已有替代路径。
- 不为兼容旧行为编写额外 wrapper，除非该 wrapper 是当前阶段的明确交付物。

## 4. 开发流程

1. 定位：先读 `docs/tech/todo-index.md`，再读本次批次所属专题文件；如涉及待办治理，读取 `docs/tech/todo-governance.md`。
2. 定界：说明本批次的目标、非目标、允许破坏的旧行为和验证方式。
3. 记录：架构、公共接口、CLI 行为、数据模型或模板协议变化，记录到 `docs/tech/*.md`。
4. 实施：按目标架构修改代码、测试和文档；旧测试阻碍重构时应更新或删除。
5. 验证：验证新目标行为，不强制验证旧入口不退化。
6. 收尾：更新路线图批次状态，提交本地 commit，不主动 push。

### 4.1 批次化执行规则

- 默认以路线图中的 Batch 为开发单位，不再以单个 `T001/T002` 细项作为独立开发单位。
- 一个 Batch 应合并强相关的模型、模板、CLI、验证和文档工作，目标是一次交付可运行的纵向能力。
- 每个 Batch 开始前只做一次边界说明；开发中不为每个细项重复规划、验证和提交。
- 每个 Batch 通常使用 1-2 个本地 commit：一个实现提交，必要时再加一个文档或产物提交；只有风险明显不同的子能力才拆更多 commit。
- 路线图中的旧细项可作为范围说明或历史记录，但不再作为执行门槛。完成状态以 Batch checkbox 为准。
- 用户明确要求“小步提交”“只做一个点”或紧急 hotfix 时，可以临时回到单切片流程。

## 5. 验证策略

- TypeScript 代码变更默认运行 `npm run build`。
- 行为变更运行相关 `vitest` 单测；必要时补充新的 smoke 测试。
- CLI 重构只 smoke 新目标命令；旧命令仅在当前阶段声明兼容时才 smoke。
- 生成产物、模板或命令 renderer 变化时，运行 `npm run build:commands` 或相关 manifest 检查。
- Batch 收尾时按影响面运行组合验证；大阶段收尾时再运行 `npm run verify`。
- 文档-only 变更至少运行 `git diff --check`，并人工检查路径、任务状态和术语一致。

## 6. 文档规则

- 重要决策优先记录在 `docs/tech/`，文件名使用英文 kebab-case。
- 路线图任务完成后，同步当前专题 roadmap、`docs/tech/todo-index.md` 和 `docs/tech/todo-archive.md`；full-refactor 历史路线只在相关批次确实被补充时更新。
- Batch 完成后必须归档到 `docs/tech/todo-archive.md`；若属于 full-refactor 主路线或补充批次，同时记录到 `docs/tech/archive/full-refactor/full-refactor-completed.md`。
- README 只在用户可见能力已经实现或旧说明已经误导时更新。
- 不把尚未实现的目标写成已完成能力。
- 不再引用 `novel-sdd.md` 作为开发依据；如需引用，只能作为产品理念背景。

## 7. 兼容与迁移

- 兼容是阶段任务，不是默认要求。
- 当任务要求删除旧入口时，应同时更新测试、文档和错误提示。
- 当任务要求保留兼容期时，应明确兼容范围、弃用提示和后续删除条件。
- 迁移说明只描述真实可执行路径，不承诺未实现的自动迁移能力。

## 8. Skill 与资料使用

- 归档 skill 只在用户明确要求或任务确实需要时加载，并在最终说明中写明使用了哪些 skill。
- 对外部框架、库、CLI API 的新写法，必要时查阅官方文档；纯内部类型、重命名和领域模型不强制联网。
- 非必要不联网；路线图、仓库代码和项目 SDD 是本项目重构的主要事实源。

## 9. 提交规则

- 完成明确代码或文档修改并通过必要验证后，创建本地 commit。
- commit message 使用中文，描述本次切片的用户可见结果或架构结果。
- 不主动 push。
